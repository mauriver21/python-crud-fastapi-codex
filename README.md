# FastAPI Codex Skills

This repository develops reusable Codex skills for creating and extending
Python 3.12 FastAPI projects. The skills contain the instructions, deterministic
generators, reference contracts, and templates Codex needs to reproduce project
conventions consistently.

The root FastAPI application is a reference and validation fixture. It
demonstrates the architecture generated by the scaffold skill; it is not the
primary deliverable of this repository.

## Skills

| Skill | Purpose |
| --- | --- |
| `fastapi-scaffold` | Reproduce the complete reference user CRUD API with FastAPI, PostgreSQL, SQLModel, JWT authentication, custom migrations, and integration tests. |
| `fastapi-configure-sqlmodel-database` | Add typed PostgreSQL and SQLModel infrastructure with Alembic migrations to a compatible FastAPI project. |
| `fastapi-add-resource-layers` | Generate a CRUD resource with schemas, models, repositories, services, routes, and API tests in a compatible layered project. |

Each skill lives under `.agents/skills/<skill-name>/` and may include:

```text
<skill-name>/
├── SKILL.md           # Trigger description, workflow, and guardrails
├── agents/
│   └── openai.yaml    # Codex UI metadata and default prompt
├── assets/            # Files copied into generated projects
├── references/        # Contracts and task-specific guidance
└── scripts/           # Deterministic generators and validators
```

## Using the skills

Invoke a skill by name in Codex:

```text
$fastapi-scaffold
```

The scaffold skill asks for an explicit destination path before inspecting or
writing files. It can generate into an empty directory or, when explicitly
requested, into an existing Git repository without creating a nested
repository:

```bash
python3.12 .agents/skills/fastapi-scaffold/scripts/scaffold_project.py \
  <destination> \
  --project-name <project-name> \
  --dry-run
```

Review the preview before running the same command without `--dry-run`. Add
`--existing-repository` only when the destination is already a Git repository.
The generator rejects conflicting target files instead of overwriting them.

The database and resource skills can be invoked independently for compatible
projects:

```text
$fastapi-configure-sqlmodel-database
$fastapi-add-resource-layers
```

Read each skill's `SKILL.md` before using its scripts. Their supported
architectures and guardrails are intentionally explicit.

## Reference application

The project at the repository root mirrors the complete application bundled in
`.agents/skills/fastapi-scaffold/assets/project-template`. It is used to
verify:

- flat `src/` imports without `__init__.py`;
- routes, controllers, repositories, models, and interfaces;
- JWT authentication middleware;
- PostgreSQL and SQLModel integration;
- the custom `db-migrate` command;
- controller-level integration tests.

Install its dependencies in an isolated Python 3.12 environment:

```bash
python3.12 -m venv .venv
source .venv/bin/activate
pip install -e ".[test]"
```

Create separate ignored `.env.dev` and `.env.test` files from
`.env.example`. Never point the test environment at development or production
data because the integration suite deletes non-fixture user records.

Apply migrations and run tests only after configuring dedicated PostgreSQL
databases:

```bash
APP_ENV=development ./.venv/bin/db-migrate upgrade
APP_ENV=test ./.venv/bin/db-migrate upgrade
./.venv/bin/pytest
```

## Developing a skill

1. Update the skill's reusable assets, scripts, references, and `SKILL.md`.
2. Keep generated output deterministic and reject file conflicts.
3. Preview generators against a temporary destination.
4. Validate the skill metadata:

   ```bash
   python3.12 ~/.codex/skills/.system/skill-creator/scripts/quick_validate.py \
     .agents/skills/<skill-name>
   ```

5. Run the generated project's installation, import, migration, and test checks
   appropriate to the change.

Do not store credentials, virtual environments, caches, generated build output,
or local database state in a skill.
